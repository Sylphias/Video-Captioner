# Phase 9.1: Transcription & Diarization Upgrade - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate the transcription and diarization pipelines from Apple Silicon CPU (Mac Mini) to NVIDIA RTX 4080 GPU on Windows. Replace faster-whisper with Parakeet TDT (fallback: WhisperX), replace pyannote diarization with NeMo MSDD. Clean platform break — Windows-only going forward. Existing transcript.json output format preserved (extend-only).

</domain>

<decisions>
## Implementation Decisions

### Transcription model
- **D-01:** Primary model is NVIDIA Parakeet TDT on CUDA — research and validate first via spike
- **D-02:** Fallback model is WhisperX if Parakeet word timestamps are insufficient or integration proves problematic
- **D-03:** Validation criteria: word-level timestamps at least as accurate as current faster-whisper large-v3, runs on CUDA with RTX 4080

### Diarization model
- **D-04:** Replace pyannote with NeMo MSDD (Multi-Scale Diarization Decoder) on CUDA
- **D-05:** NeMo ecosystem pairs with Parakeet — investigate shared dependencies and pipeline integration

### Platform
- **D-06:** Windows-only going forward — drop all Mac-specific code and paths
- **D-07:** Remove hardcoded `/opt/homebrew/bin/ffmpeg` — use `ffmpeg` on PATH (Windows)
- **D-08:** Python venv path changes from `.venv/bin/python` to `.venv\Scripts\python.exe`
- **D-09:** Device selection hardcoded to CUDA — no CPU fallback needed

### Output format
- **D-10:** Transcript.json format preserved: `{language, words: [{word, start, end, confidence, speaker}]}`
- **D-11:** New fields allowed (e.g. punctuation, sentence boundaries from Parakeet) but existing fields must stay unchanged
- **D-12:** Diarization output still enriches transcript.json in-place with per-word speaker labels

### Claude's Discretion
- NeMo/Parakeet Python dependency management (pip, conda, or other)
- Exact validation spike structure and pass/fail criteria
- Whether transcription and diarization scripts stay as separate .py files or merge
- Windows justfile / setup commands
- Progress reporting format adaptation (JSON-line protocol stays, internal messages may change)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `transcription.ts` / `diarization.ts` — subprocess spawn + JSON-line progress pattern. Same pattern works for new Python scripts, just update paths and args.
- JSON-line progress protocol (`{type: "progress", percent: N}`) — frontend SSE already consumes this; new scripts should emit same format.
- `justfile` setup-python recipe — needs rewrite for Windows + NeMo/Parakeet deps.

### Established Patterns
- Python subprocess per job (not persistent daemon) — spawn, run, exit. Simple and proven.
- Progress via stdout JSON lines, errors via stderr + exit code.
- Transcript enrichment in-place (diarize.py reads + overwrites transcript.json).

### Integration Points
- `packages/backend/src/routes/presets.ts` triggers transcription/diarization via service modules
- Frontend hooks (`useTranscribe`, `useDiarize`) consume SSE progress events
- `subtitleStore.ts` parses transcript.json into SessionWord/SessionPhrase structures
- `SubtitleOverlay.tsx` renders from CompositionPhrase — fully decoupled from transcription backend

</code_context>
