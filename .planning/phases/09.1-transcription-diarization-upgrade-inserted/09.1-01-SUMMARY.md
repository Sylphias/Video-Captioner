## Decisions Made

- **Option B (WSL):** NeMo's `triton` dependency has no Windows wheel; native Windows install fails at that step. WSL provides full Linux environment sharing the same NVIDIA GPU via CUDA-on-WSL drivers. User confirmed this based on research before the spike was run.
- **WSL_PYTHON = '/root/.venv-wsl/bin/python3':** Fixed absolute Linux path. Using `/root/` avoids NTFS filesystem permission issues that arise when creating a Python venv on the Windows-side mounted drive (`/mnt/f/...`) in WSL — venv activation symlinks require Linux permissions.
- **toWslPath() at spawn time:** Path conversion happens in the service layer just before spawn, keeping jobs, routes, and store fully Windows-path-aware. Only the subprocess boundary converts.
- **Retained setup-python (WhisperX native Windows):** Kept as reference/fallback recipe. Plan 02 determines whether it stays or is removed.

**2. [Rule 2 - Missing Critical] WSL venv placed at /root/.venv-wsl rather than repo-relative path**
- **Found during:** Task 2 implementation
- **Issue:** Plan implied `.venv-wsl` relative to repo root. Creating a Python venv on a Windows NTFS mount (accessed from WSL at `/mnt/f/...`) fails — NTFS does not support Linux file permissions needed for venv symlinks; pip scripts and activate fail.
- **Fix:** Placed venv at `/root/.venv-wsl` (pure Linux filesystem inside WSL). justfile recipe uses `wsl bash -c "python3 -m venv /root/.venv-wsl && ..."`.
- **Files modified:** justfile, transcription.ts (WSL_PYTHON constant), diarization.ts (WSL_PYTHON constant)
- **Committed in:** cfa645e
