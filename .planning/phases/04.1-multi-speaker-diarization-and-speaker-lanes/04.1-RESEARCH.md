# Phase 04.1: Multi-Speaker Diarization and Speaker Lanes - Research

**Researched:** 2026-03-04
**Domain:** Speaker diarization (Python), word-level speaker assignment algorithm, TypeScript type propagation, Zustand store extension, React speaker lane UI
**Confidence:** HIGH for diarization library selection and integration algorithm; MEDIUM for HuggingFace-free path (diarize library is recent); LOW for pyannote 4.x + Apple Silicon MPS (conflicting reports)

## Summary

Phase 04.1 adds "who spoke when" to every word in the transcript. The work has four layers: (1) run diarization in the Python subprocess and emit speaker labels per word in the JSON output, (2) propagate a `speaker` field through the TypeScript type system (shared types → grouping → store → composition), (3) render speaker lanes in the transcript editor UI, and (4) allow manual reassignment of speaker labels.

The most consequential decision is **which Python diarization library to use**. There are two viable options with meaningfully different tradeoffs:

- **pyannote-audio 3.3.2** (`pip install pyannote-audio==3.3.2`): Industry-standard, well-documented, ~10-19% Diarization Error Rate (DER) on benchmarks. Requires a HuggingFace account and a read token for model download. Works on Apple Silicon CPU with PyTorch (`torch>=2.0.0` — no upper bound, so it coexists with the project's existing `torch 2.10.0`). MPS acceleration on Apple Silicon is technically possible but has known accuracy regressions (wrong timestamps, single-speaker output); CPU is the safe path. Real-time factor on CPU is roughly 1x–2x (a 10-minute video takes 30–90 seconds to diarize). The `speaker-diarization-3.1` model is used with this version.
- **diarize 0.x** (`pip install diarize`): Newer, no token, no account signup, Apache 2.0 license, CPU-only by design. Uses ONNX Runtime with Silero VAD + WeSpeaker ResNet34-LM embeddings + spectral clustering. Reports ~10.8% DER on VoxConverse and ~8x real-time processing speed on CPU. ONNX Runtime already exists in the project's `.venv` (version 1.24.2, which includes arm64 macOS wheels), so no new binary conflict arises. Output API: `result.segments` is a list of objects with `start`, `end`, `speaker` fields.

The architecture for integration is identical for both libraries: run diarization first (either as a second Python script or within the same `transcribe.py` script), collect speaker-labeled segments (each with `start`, `end`, `speaker`), then for each Whisper word, find the diarization segment with the greatest temporal overlap and assign its speaker label. This produces a `TranscriptWord` with an optional `speaker` field.

The frontend changes are additive, not replacements: `SessionWord` and `SessionPhrase` get a `speaker?: string` field; the store gains a `speakerNames` map for display labels; the `TranscriptEditor` gets a speaker lane visual (color-coded left border per phrase, a speaker badge per phrase row, and a dropdown to reassign speaker).

**Primary recommendation:** Use `pyannote-audio==3.3.2` with `speaker-diarization-3.1` for accuracy and ecosystem maturity. Require a HuggingFace token configured as an environment variable (`HUGGINGFACE_TOKEN`). If a "zero-signup" path is important, `diarize` is a viable alternative with comparable accuracy. Run diarization as a second step inside `transcribe.py` (both in the same `.venv` subprocess), emitting a combined JSON with `speaker` on each word. Do not split into two subprocess calls — file I/O coordination is not worth the complexity.

### Pattern 1: Two-Step Python Pipeline (transcribe then diarize in one script)

**What:** `transcribe.py` runs faster-whisper first, collects all words with timestamps, then runs diarization on the same audio file, then assigns speaker labels to words by temporal overlap, then writes the enriched JSON.

**When to use:** Always — running both steps in one subprocess call avoids file handoff complexity and allows shared in-memory state.

**Example (pyannote path):**
```python
# scripts/transcribe.py — new diarization section after words are collected

import os
from pyannote.audio import Pipeline
import torch

def assign_speakers(words: list[dict], audio_path: str, hf_token: str) -> list[dict]:
    """Run pyannote diarization and assign speaker to each word by max overlap."""
    pipeline = Pipeline.from_pretrained(
        "pyannote/speaker-diarization-3.1",
        use_auth_token=hf_token
    )
    # Stay on CPU — MPS has known accuracy regressions on Apple Silicon
    pipeline.to(torch.device("cpu"))

    diarization = pipeline(audio_path)

    # Build list of (start, end, speaker) from pyannote output
    segments = [
        (turn.start, turn.end, speaker)
        for turn, _, speaker in diarization.itertracks(yield_label=True)
    ]

    # Assign each word to the speaker with the greatest temporal overlap
    for word in words:
        word_start, word_end = word["start"], word["end"]
        best_speaker = None
        best_overlap = 0.0

        for seg_start, seg_end, speaker in segments:
            overlap = min(word_end, seg_end) - max(word_start, seg_start)
            if overlap > best_overlap:
                best_overlap = overlap
                best_speaker = speaker

        word["speaker"] = best_speaker  # None if no overlap found

    return words
```

**Example (diarize path):**
```python
from diarize import diarize as run_diarize

def assign_speakers(words: list[dict], audio_path: str) -> list[dict]:
    result = run_diarize(audio_path)
    segments = [(seg.start, seg.end, seg.speaker) for seg in result.segments]

    for word in words:
        word_start, word_end = word["start"], word["end"]
        best_speaker = None
        best_overlap = 0.0
        for seg_start, seg_end, speaker in segments:
            overlap = min(word_end, seg_end) - max(word_start, seg_start)
            if overlap > best_overlap:
                best_overlap = overlap
                best_speaker = speaker
        word["speaker"] = best_speaker

    return words
```

### Pattern 2: Type Propagation (shared-types → grouping → store → composition)

**What:** Add `speaker?: string` to `TranscriptWord` in shared-types. Propagate to `SessionWord`, `SessionPhrase` (as `dominantSpeaker?: string`), and optionally to `TranscriptPhrase` for composition.

**When to use:** The `speaker` field must flow through the whole pipeline so that (a) the editor can render lanes and (b) the composition can optionally display speaker names.

**Example:**
```typescript
// packages/shared-types/src/index.ts
export interface TranscriptWord {
  word: string
  start: number
  end: number
  confidence: number
  speaker?: string   // "SPEAKER_00", "SPEAKER_01", etc. — undefined if single-speaker or no diarization
}

// packages/frontend/src/lib/grouping.ts
export interface SessionWord {
  word: string
  start: number
  end: number
  confidence: number
  speaker?: string   // propagated from TranscriptWord
}

export interface SessionPhrase {
  words: SessionWord[]
  isManualSplit: boolean
  dominantSpeaker?: string  // most common speaker label among words in phrase
}
```

### Pattern 3: Speaker Names Map in Store

**What:** The store holds a `speakerNames: Record<string, string>` mapping raw labels (`"SPEAKER_00"`) to display names (`"Alice"`). A `renameSpeaker` action updates this map. A `reassignWordSpeaker` action changes the speaker field on a word (and rebuilds phrase `dominantSpeaker`).

**When to use:** This is the manual override mechanism. It keeps raw speaker IDs stable while allowing user-friendly display names.

**Example:**
```typescript
// In subtitleStore.ts — new fields and actions
interface SubtitleStore {
  // ... existing fields ...
  speakerNames: Record<string, string>  // e.g., { SPEAKER_00: "Alice", SPEAKER_01: "Bob" }

  // Actions
  renameSpeaker: (speakerId: string, name: string) => void
  reassignWordSpeaker: (wordIndex: number, speakerId: string) => void
}
```

### Anti-Patterns to Avoid

- **Running diarization and transcription as two separate subprocess calls:** Adds file handoff complexity (passing intermediate JSON), doubles process startup overhead, and makes progress reporting harder.
- **Using pyannote-audio 4.x (community-1):** Pins `torch==2.8.0` exactly, which conflicts with the project's existing `torch 2.10.0` in `.venv`. Would require a complete venv rebuild and risk breaking faster-whisper.
- **Enabling MPS (`pipeline.to(torch.device("mps"))`) on Apple Silicon:** Multiple users report drastic accuracy degradation — entire recordings attributed to a single speaker — even when MPS does not throw an error. Stick to CPU for pyannote.
- **Storing `speakerNames` in Python output:** The transcript JSON should only contain raw IDs (`"SPEAKER_00"`). Display names are session state, not ground truth — store them in Zustand only.
- **Rebuilding `SessionPhrase[]` on every speaker reassignment:** Speaker reassignment only changes the `speaker` field on one word and the `dominantSpeaker` on the affected phrase. Avoid triggering a full `buildSessionPhrases()` rebuild for speaker-only changes (same pitfall as text-only edits in Phase 4).
- **Requiring diarization always:** Diarization adds 30–90 seconds to the workflow. The feature must be opt-in (a "Diarize speakers" toggle or button, separate from "Transcribe"), not triggered automatically.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Speaker segmentation model | Custom VAD + embedding model | pyannote/speaker-diarization-3.1 or diarize | Neural segmentation has dozens of edge cases; speaker embedding is a research area |
| Word-to-speaker temporal matching | Ad hoc midpoint assignment | Max-overlap algorithm (documented above) | Midpoint assignment fails at speaker transitions where word spans the boundary |
| Speaker count estimation | Rule-based silence analysis | GMM BIC (in diarize) or pyannote's built-in agglomerative clustering | Speaker count estimation is a known-hard problem; existing implementations are well-tested |
| Speaker color palette | Hand-picked hardcoded colors | CSS custom property index pattern (above) | With 8 colors, index cycling covers the realistic 2–6 speaker range cleanly |

**Key insight:** The word-level speaker assignment is deceptively simple — 15 lines of max-overlap code — but the correctness details matter: use cumulative overlap per speaker (not first-match), handle the case where a word has no overlapping segment (assign `null`/`undefined`), and handle overlapping speech regions where two speakers are labeled simultaneously.

## Common Pitfalls

### Pitfall 1: pyannote 4.x Torch Version Conflict
**What goes wrong:** `pip install pyannote-audio` installs 4.0.x which pins `torch==2.8.0` exactly. The project's `.venv` has `torch 2.10.0`. pip will downgrade torch to 2.8.0, which may break `ctranslate2 4.7.1` (requires torch>=2.4).
**Why it happens:** pyannote 4.0.2+ introduced an exact pin after experiencing segmentation faults in `torchcodec`.
**How to avoid:** Pin to `pyannote-audio==3.3.2`. Version 3.3.2 requires only `torch>=2.0.0` (no upper bound). Confirmed compatible with `torch 2.10.0`.
**Warning signs:** pip install output shows `Uninstalling torch-2.10.0`, or `ctranslate2` starts failing.

### Pitfall 2: HuggingFace Token Not Available at Runtime
**What goes wrong:** `transcribe.py` calls `Pipeline.from_pretrained(...)` with `use_auth_token=None` (env var missing) — pipeline raises `OSError: You are trying to access a gated repo`.
**Why it happens:** pyannote/speaker-diarization-3.1 is a gated model; the token is required for download and is checked on each instantiation unless the model is cached locally.
**How to avoid:** Read `HUGGINGFACE_TOKEN` from environment in the Python script; raise a descriptive error if missing and diarization was requested. The backend must pass this env var to the subprocess. Models cache locally after first download (typically in `~/.cache/huggingface/hub/`) — subsequent runs work offline.
**Warning signs:** Process exits with non-zero code and stderr contains "gated repo" or "not in the authorized list".

### Pitfall 3: MPS Accuracy Regression on Apple Silicon
**What goes wrong:** Diarization completes but assigns all words to `SPEAKER_00`, or produces wildly wrong speaker boundaries.
**Why it happens:** PyTorch MPS does not implement all FFT operations pyannote needs; some fall back to CPU silently, producing incorrect intermediate tensors that lead to wrong clustering.
**How to avoid:** Always use `torch.device("cpu")` for pyannote on Apple Silicon. Do not use `pipeline.to("mps")`.
**Warning signs:** All words have the same `speaker` value; or speaker boundary timestamps are obviously wrong compared to audio content.

### Pitfall 4: Speaker Rebuild Triggering Full Phrase Rebuild
**What goes wrong:** User clicks "reassign speaker" for one word; the store action calls `buildSessionPhrases()`, which recomputes all phrase boundaries, losing manual splits or causing visible flash.
**Why it happens:** Calling `buildSessionPhrases()` is the easy path for any store update. But speaker labels are word-level metadata, not boundary-affecting data — phrase boundaries should not change when a speaker label changes.
**How to avoid:** `reassignWordSpeaker` action updates only `session.words[i].speaker` and recomputes `phrase.dominantSpeaker` inline (iterate phrase words, find most common speaker). Never calls `buildSessionPhrases()`.
**Warning signs:** Manual phrase splits disappear after a speaker reassignment.

### Pitfall 5: Diarization Triggered Automatically on Every Transcription
**What goes wrong:** Transcription + diarization always runs together; diarization adds 30–90s overhead on single-speaker videos (the common case for the app's current use).
**Why it happens:** It's tempting to bundle everything in one step since the audio file is available.
**How to avoid:** Diarization must be opt-in. Add a `--diarize` flag to `transcribe.py` (or add a separate `diarize` endpoint in Fastify). The UI exposes a "Detect speakers" button that triggers a separate diarization job after transcription.
**Warning signs:** Every transcription is noticeably slower even for solo-speaker content.

### Pitfall 6: `dominantSpeaker` Algorithm Choosing Wrong Speaker
**What goes wrong:** A phrase with words `[SPEAKER_00, SPEAKER_00, SPEAKER_01]` gets labeled `SPEAKER_00` correctly, but a phrase with `[SPEAKER_00, SPEAKER_01]` breaks ties arbitrarily.
**Why it happens:** Dominant speaker by word count doesn't account for word duration or silence gaps.
**How to avoid:** Count words (not duration) for simplicity; ties go to the speaker of the first word in the phrase. Document this clearly. Speaker boundaries are at phrase level, so ties are rare — the phrase algorithm already splits on silence gaps which usually align with speaker transitions.

### Word-level speaker assignment (max-overlap algorithm)
```python
# Source: Verified from WhisperX implementation (deepwiki.com/m-bain/whisperX/3.4-speaker-diarization)
# and scalastic.io/en/whisper-pyannote-ultimate-speech-transcription/
def assign_speakers_to_words(words: list[dict], segments: list[tuple]) -> list[dict]:
    """
    words: [{"word": str, "start": float, "end": float, ...}, ...]
    segments: [(seg_start, seg_end, speaker_id), ...]
    """
    for word in words:
        word_start = word["start"]
        word_end = word["end"]
        overlap_by_speaker: dict[str, float] = {}

        for seg_start, seg_end, speaker in segments:
            overlap = min(word_end, seg_end) - max(word_start, seg_start)
            if overlap > 0:
                overlap_by_speaker[speaker] = overlap_by_speaker.get(speaker, 0) + overlap

        if overlap_by_speaker:
            word["speaker"] = max(overlap_by_speaker, key=overlap_by_speaker.get)
        else:
            word["speaker"] = None  # word in gap between speaker segments

    return words
```

### dominantSpeaker computation
```typescript
// In grouping.ts — compute dominantSpeaker for a phrase
function computeDominantSpeaker(words: SessionWord[]): string | undefined {
  const counts: Record<string, number> = {}
  for (const w of words) {
    if (w.speaker) counts[w.speaker] = (counts[w.speaker] ?? 0) + 1
  }
  const speakers = Object.keys(counts)
  if (speakers.length === 0) return undefined
  // Tie-break: first word's speaker wins
  return speakers.reduce((a, b) => counts[a] >= counts[b] ? a : b)
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual speaker annotation | pyannote neural diarization | ~2021 | 10–19% DER vs 30%+ manual |
| onnxruntime dependency in pyannote | Pure PyTorch (3.1+) | Oct 2023 | Removed Mac incompatibility from onnxruntime-gpu |
| pyannote 3.x flexible torch | pyannote 4.x pins torch==2.8.0 | ~Jan 2025 | Breaking change for projects with other torch consumers |
| Segment-level speaker assignment | Word-level max-overlap | 2023 (WhisperX) | Accurate speaker label per word even at speaker transitions |

**Deprecated/outdated:**
- `use_auth_token` kwarg in pyannote: Deprecated in favor of `token` kwarg in newer versions. Both work in 3.3.2; use `token` for forward compatibility.
- pyannote 2.x: Uses onnxruntime-gpu which breaks on Apple Silicon; upgrade to 3.3.2.
- Midpoint-based speaker assignment: Assigns a word to whatever speaker was talking at its midpoint. Fails when a word spans a speaker boundary. Use max-overlap instead.

## Open Questions

1. **Should diarization run in the same `transcribe.py` script or a separate `diarize.py`?**
   - What we know: Same script avoids file I/O handoff and doubles as a single subprocess to manage. Separate script allows independent triggering (re-diarize without re-transcribing).
   - What's unclear: Will the user want to re-diarize after editing timestamps? (Yes — editing timestamps could change word-to-segment overlap; re-diarize makes sense after major edits.)
   - Recommendation: Two separate scripts — `transcribe.py` (current) and `diarize.py` (new). The backend has separate endpoints. The UI has a "Detect speakers" button. Re-diarization on existing transcript is a valid use case.

2. **pyannote 3.3.2 vs `diarize` library — which to recommend?**
   - What we know: pyannote 3.3.2 has better documentation, larger community, and model variants. `diarize` library has no HuggingFace account requirement, ONNX-based so already in venv, claims 8x realtime speed, similar DER (~10.8% vs 11.2%).
   - What's unclear: `diarize` library is newer (March 2026) with less production track record.
   - Recommendation: Research recommends pyannote 3.3.2 as primary, with `diarize` as a clearly documented alternative if HuggingFace friction is a concern. The planner should choose one and document it as the standard for this phase.

3. **Does the composition (Remotion) need to render speaker names?**
   - What we know: `SubtitleComposition` receives `phrases: TranscriptPhrase[]`. Adding `speaker?: string` to `TranscriptPhrase` is low-cost.
   - What's unclear: The phase description mentions "speaker lanes in the transcript editor UI" but not the composition. Rendering speaker names in the video output may be out of scope.
   - Recommendation: Add `speaker?: string` to `TranscriptPhrase` in shared-types now (future-proofing), but do not implement speaker-label rendering in the composition in this phase. The editor lanes are the deliverable.

4. **What happens when no diarization is run (legacy transcripts)?**
   - What we know: `speaker?: string` is optional. Existing transcript JSON has no `speaker` field.
   - What's unclear: Does the UI need to gracefully degrade, or does the "Detect speakers" button only appear after transcription?
   - Recommendation: The store detects whether the loaded transcript has any `speaker` values (`transcript.words.some(w => w.speaker != null)`). If no speakers, the transcript editor renders in its existing single-speaker mode. The "Detect speakers" button is always visible after transcription.
